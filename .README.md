# AI-Powered Worker Productivity Dashboard

A full-stack, containerized application designed to ingest, process, and visualize AI-generated worker activity events from factory CCTV cameras.

## 🚀 Quick Start (Docker)

This application is fully containerized. To run it locally:

1. Ensure Docker and Docker Compose are installed.
2. Run the following command in the root directory:
   \`\`\`bash
   docker-compose up --build
   \`\`\`
3. Once the containers are running, navigate to \`http://localhost:5173\` to view the dashboard.
4. **Important:** To populate the database with meaningful data on the first run, execute the seeder route:
   \`\`\`bash
   curl -X POST http://localhost:5000/api/seed
   \`\`\`

## 🏗️ Architecture Overview

**Edge → Backend → Dashboard**

1. **Edge (Cameras):** The computer vision system acts as the data producer, sending structured JSON events via HTTP POST requests.
2. **Backend (Express.js + TypeScript):** Acts as the ingestion engine and calculation brain. It exposes RESTful APIs to ingest data and serves pre-calculated metrics to the frontend.
3. **Database (MongoDB):** A NoSQL database chosen for its flexibility with JSON payloads and high-write throughput capabilities.
4. **Frontend (React + Vite + Tailwind):** A minimalist, industrial-themed dashboard that polls the backend for pre-calculated metrics and visualizes factory utilization.

## 🧠 Metric Definitions & Assumptions

- **Active Time:** Calculated by measuring the time difference between an event and the preceding event. If the preceding event was "working", that time delta is added to the active bucket.
- **Idle Time:** Calculated similarly to Active Time, but applied when the preceding state was "idle".
- **Utilization Percentage:** `(Active Time / (Active Time + Idle Time)) * 100`
- **Assumption:** The math engine assumes events are processed chronologically based on their `timestamp` payload, not their arrival time at the server.

## 🛠️ Theoretical Architecture & Handling Edge Cases

### Handling Intermittent Connectivity, Duplicates, and Out-of-Order Timestamps

- **Intermittent Connectivity:** Edge devices (cameras) should not send HTTP requests blindly. They need a local buffer or queue (like a lightweight SQLite DB or MQTT client). If the connection drops, events are stored locally and flushed to the backend in a batch once connectivity is restored.
- **Out-of-Order Timestamps:** Because of network lag, Event B might arrive before Event A. To handle this, our backend calculation engine (`getMetrics`) does not rely on the database insertion order. Instead, it queries the database and sorts all events by the `timestamp` field before performing any time-delta calculations.
- **Duplicate Events:** To prevent counting the same event twice (e.g., if a camera retries a failed request), we would implement **Idempotency**. We can create a unique composite index in MongoDB using `{ worker_id, workstation_id, timestamp }`. If the edge sends the exact same payload twice, the database will reject the duplicate.

### Model Lifecycle (Versioning, Drift, and Retraining)

- **Model Versioning:** The API payload should be updated to include a `model_version` field (e.g., `"model_version": "v1.2.0"`). This allows the backend to tag which model generated which event, making it easier to rollback if a specific model starts producing erratic data.
- **Detecting Model Drift:** We can monitor the `confidence` score over time. If the average confidence score across the factory drops below a certain threshold (e.g., consistently below 0.70), it triggers an alert indicating that the factory environment has changed (e.g., new lighting, new uniforms) and the model is drifting.
- **Triggering Retraining:** Images that result in low-confidence scores can be automatically routed to a secure S3 bucket. A "Human-in-the-loop" pipeline can then manually label these edge-case images, adding them to the dataset for the next model retraining epoch.

### Scaling from 5 Cameras to 100+ Cameras (Multi-Site)

If we scale to hundreds of cameras, our standard Express REST API will eventually drop requests due to high load.

1. **Event Broker (Message Queue):** We would introduce **Apache Kafka** or **RabbitMQ** between the Edge and the Backend. Cameras publish events to Kafka, and the Node.js backend consumes them at its own pace. This guarantees zero data loss during high traffic spikes.
2. **Horizontal Scaling:** We would containerize the Node.js ingestion API and deploy it to a Kubernetes cluster, allowing it to auto-scale based on CPU utilization.
3. **Database Sharding:** As the MongoDB collection grows to millions of rows, we would shard the database (likely by `factory_id` or `timestamp`) to maintain fast query performance for the metrics engine.
